서울시 창업 AI 컨설턴트 (LLM Multi-Agent 기반 상권 분석 시스템)

1. 개요

서울시 25개 자치구의 상권 데이터와 LSTM 매출 예측 모델을 결합한 Multi-Agent LLM 시스템입니다. LangChain과 LangGraph를 활용하여 6개의 전문 에이전트가 협업하며, RAG 기반 문서 검색과 메타데이터 관리를 통해 신뢰성 있는 창업 컨설팅을 제공합니다.

2. 사용 기술

Python, LangChain, LangGraph, OpenAI GPT-4o, FAISS Vector Store, TensorFlow/Keras (LSTM), Pandas, NumPy, Gradio, Seoul Open API, LangSmith, ChromaDB, Git/GitHub

3. 핵심 기능

- LangChain + LangGraph: Multi-Agent 오케스트레이션
- OpenAI GPT-4o: LLM 기반 분석 및 추론
- FAISS Vector Store: 하이브리드 RAG 검색
- LSTM Deep Learning: 시계열 매출 예측
- Gradio: 인터랙티브 웹 인터페이스

4. Multi-Agent 구성

- Location Agent: 입지 분석 및 상권 특성 평가
- Competition Agent: 경쟁 현황 및 폐업률 분석
- Predictor Agent: LSTM 기반 매출 예측
- Risk Assessment Agent: 창업 리스크 종합 평가
- RAG Agent: 메타데이터 기반 문서 검색
- Report Agent: 최종 분석 보고서 생성

Troubleshooting & Lessons Learned

1. AI 할루시네이션과 데이터 검증의 중요성

문제 상황

AI는 강남구 데이터를 활용해 분석했다고 설명했지만, 실제 원본 데이터를 확인한 결과 강동구 자료가 사용되고 있었습니다. 자치구 이름에 대한 필터링이 적용되지 않아 분석 대상이 의도와 다르게 섞여 있었습니다. 또한 Open API로 가져온 데이터는 자치구가 아닌 상권 단위로만 구성되어 있어, 기존에 계획했던 자치구별 분석 구조를 그대로 적용할 수 없었습니다.

해결 과정

AI의 설명을 그대로 신뢰하지 않고, API 문서와 실제 데이터 컬럼을 직접 확인하며 분석 가능한 기준 단위를 재검토했습니다. 데이터 구조에 맞게 분석 범위를 조정하고, 자치구 기준 분석이 가능한 데이터만 선별하여 사용했습니다.

배운 점

AI는 개발 생산성을 크게 높여주지만, "데이터 수집과 검증 단계에서는 여전히 사람의 확인이 필수적"이라는 점을 깨달았습니다. 특히 AI 할루시네이션을 방지하기 위해서는 코드 결과뿐 아니라 "원본 데이터 자체를 직접 검증하는 태도"가 중요합니다.

2. 데이터 컬럼 해석 오류와 지표 정확성

문제 상황

상권 데이터를 활용해 편의점 폐업률을 산출하는 과정에서, 강동구 전체 폐업률이 300%로 계산되는 비정상적인 결과가 발생했습니다. 초기에는 점포수 컬럼을 전체 점포 수로 간주했으나, 실제 데이터 구조상 전체 점포 수는 "점포수 + 프랜차이즈 점포수의 합"으로 계산해야 했습니다. 유사업종 점포수는 비교용 지표로, 폐업률 산정에는 포함되지 않아야 하는 컬럼이었습니다.

해결 과정

행정동 단위 데이터를 기준으로 컬럼의 의미를 재정의하고, 전체 점포 수를 점포수 + 프랜차이즈 점포수로 재계산하여 폐업률 산식을 수정했습니다. 그 결과 강일동 주민센터 인근 편의점의 폐업률이 실제 값인 약 17%로 정상적으로 산출되는 것을 확인했습니다.

배운 점

AI나 모델 이전에 "데이터 컬럼의 의미와 산정 기준을 정확히 이해하는 것이 가장 중요하다"는 점을 배웠습니다. AI는 주어진 데이터를 그대로 계산할 뿐, 데이터 구조의 오류를 스스로 인지하지 못하기 때문에 결과를 해석하고 검증하는 역할은 결국 개발자의 책임입니다.

3. 비율 지표의 집계 방식 오류

문제 상황

상권 데이터를 기반으로 구 단위 폐업률을 산출하는 과정에서, AI가 개별 상권의 폐업률을 단순 합산하거나 평균내어 구 전체 폐업률을 계산하는 오류가 발생했습니다. 폐업률은 비율 지표이기 때문에, 상권별 규모(점포 수)를 고려하지 않은 채 수치를 결합하면 실제보다 과도하게 왜곡된 수치가 도출됩니다.

해결 과정

상권별 폐업률을 직접 사용하는 방식을 중단하고, 점포수 + 프랜차이즈 점포수를 기준으로 전체 점포 수를 산정한 뒤 상권별 폐업 점포 수를 합산하여 구 단위 폐업률을 재계산하도록 계산식을 수정했습니다.

배운 점

AI는 계산을 자동화해 줄 수는 있지만, "지표의 정의와 수학적 타당성까지 스스로 판단하지는 못한다"는 점을 체감했습니다. 지표의 의미와 계산 방식이 합리적인지 반드시 검증하는 습관을 갖게 되었습니다.

4. RAG 메타데이터 관리의 중요성

문제 상황

RAG 기반 시스템을 설계하는 과정에서, 문서 출처와 사용 목적이 명확히 관리되지 않아 답변의 근거를 추적하기 어렵고, 데이터가 늘어날수록 시스템 일관성이 떨어지는 문제가 발생했습니다. 여러 에이전트와 도메인 문서를 함께 사용하는 구조에서는 어떤 정보가 어떤 목적에 사용되는지 명확히 구분되지 않으면 응답 신뢰도가 낮아졌습니다.

해결 과정

단순히 문서를 추가하는 방식이 아니라, 각 문서의 "출처, 사용처, 도메인, 최신성"을 메타데이터로 정의해 관리하는 구조로 접근했습니다. 에이전트별로 필요한 정보만 선택적으로 활용할 수 있었고, 도메인별 RAG 컬렉션을 분리하여 검색 범위와 컨텍스트를 명확히 했습니다.

배운 점

RAG 시스템의 품질은 모델 성능보다도 "데이터를 어떻게 설명하고, 어떻게 연결하느냐에 달려 있다"는 점을 배웠습니다. 메타데이터는 단순한 부가 정보가 아니라, 시스템의 신뢰성과 확장성을 결정하는 핵심 설계 요소입니다.

5. RAG 공간 단위 불일치 문제

문제 상황

RAG 기반 데이터 활용 과정에서 유동인구 데이터가 행정동, 자치구, 상권배후지 등 서로 다른 공간 단위로 분리되어 있어 모델 입력 기준과 불일치하는 문제가 발생했습니다. 모델은 자치구 단위를 기준으로 설계되어 있었기 때문에, 행정동 단위 데이터를 그대로 사용할 경우 분석 결과가 왜곡될 가능성이 있었습니다.

해결 과정

행정동-자치구-상권배후지 간의 소속 관계를 정의할 수 있는 외부 자료를 추가로 확보하고, 각 데이터에 관계 정보를 메타데이터 형태로 부여했습니다. 해당 메타데이터를 활용해 데이터를 자치구 단위로 통합하여, 모델 입력에 일관되게 사용할 수 있는 데이터셋을 구성했습니다.

배운 점

RAG 환경에서 중요한 것은 단순히 많은 문서를 수집하는 것이 아니라, "데이터 간의 관계를 명확히 정의하고 검색 가능한 구조로 설계하는 것"임을 체감했습니다.

6. 소비 패턴 분석의 비교 기준 설계

문제 상황

자치구별 소비 지출 데이터를 분석한 결과, AI가 특정 자치구에서 식료품 지출 비율이 높다는 이유로 식료품 업종 창업을 추천했습니다. 그러나 서울시 전 자치구를 비교 분석한 결과, 식료품 지출 비율은 모든 자치구에서 50% 이상으로 나타났으며, 이는 특정 지역의 특성이 아니라 도시 소비 구조의 공통 특성이었습니다.

해결 과정

업종 추천의 기준을 절대 비율이 아닌 "서울시 전체 평균 대비 상대적 차이"로 재설계했습니다. 자치구별 업종 지출 비율을 서울시 평균과 비교하여, 특정 업종의 지출 비중이 평균보다 유의미하게 높은 경우에만 해당 업종을 창업 추천 대상으로 선정하도록 로직을 수정했습니다.

배운 점

데이터 분석에서 중요한 것은 수치 자체보다 "어떤 기준과 관점으로 비교하느냐"라는 점을 배웠습니다. AI는 패턴을 빠르게 계산해 줄 수 있지만, 분석 목적에 맞는 비교 기준을 설정하는 역할은 결국 사람의 몫입니다.

7. LangChain 생태계 의존성 충돌 문제

문제 상황

LLM 에이전트 구축 과정에서 LangChain 생태계의 잦은 버전 업데이트로 인해 패키지 간 의존성 충돌 문제가 발생했습니다. LangSmith 설치 과정에서 ResolutionImpossible 에러가 반복적으로 발생하며, 단순히 최신 라이브러리를 설치하는 방식으로는 문제를 해결할 수 없었습니다.

해결 과정

에러 원인을 분석한 결과 각 라이브러리 간의 호환성 문제에서 비롯된 것임을 파악했고, 의존성 그래프를 기준으로 버전 조합을 재정리하여 개발 환경을 새롭게 구성했습니다. 가상환경을 분리하고 특정 버전을 명시함으로써 환경 재현성(Reproducibility)을 확보했습니다.

배운 점

복잡한 LLM 워크플로우에서는 모델 설계뿐 아니라, 라이브러리 관리와 환경 구성까지 포함한 전체 시스템 관점의 접근이 필요하다는 것을 배웠습니다. 이후에는 초기 설계 단계부터 환경 안정성을 고려하는 습관을 갖게 되었습니다.
